# Today Routines

## Что пользователь делает на экране Today

Основной сценарий:

1. Открывает `/today`
2. Видит список ежедневных рутины
3. Быстро отмечает выполнение (обычно 1 клик)
4. Получает обновление прогресса

## Универсальная модель “routine → complete”

На фронтенде используется модель:

- `Routine` (`entities/routine/types.ts`)
  - `type` — тип активности (например `steps`, `workout`, `water`, `sleep`, `custom`)
  - `targetValue` — цель
  - `unit` — единицы
- `ActionCompletion` (`entities/action/types.ts`)
  - `date` — день выполнения
  - `value` — числовое значение или агрегированная метрика

Для boolean-типов (например “выпил воду”) в домене обычно хранится:

- либо `value = 1/0`
- либо отдельный тип завершения

На уровне UI это можно свести к единому комлектору “значение/состояние → отправка в backend”.

## Типы рутины (что поддерживаем в дизайне)

- шаги (`steps`)
- тренировка (`workout`)
- вода (`water`)
- сон (`sleep`)
- “custom”

## Профили/цели и прогресс

UI должен вычислять прогресс:

- `progress = completedValue / targetValue` (с учетом лимитов/каппа, которые задаются в доменных правилах)
- отображать streak (текущий — можно по данным history)

## Анти-абьюз на UI-уровне (как минимум)

UI-ограничения:

- запрет повторного “complete” без явного подтверждения
- предохранители для необычных частот (например подсказка “это выглядит как ошибка”)
- отображение статуса отправки (loading/failed)

Реальная защита (верификация) — на backend. UI только уменьшает вероятность ошибок.

## Offline-first (закладываемое поведение)

Требование:

- пользователь может отметить выполнение даже при нестабильной сети
- данные должны встать в очередь и синхронизироваться при восстановлении сети

Реализация обычно строится так:

- локальная очередь (например в IndexedDB)
- RTK Query/Service Worker retry strategy
- background sync (опционально, в зависимости от выбранной стратегии PWA)

Текущий код сегодня — UI каркас и “точка подключения” под реальные запросы.

## Как это подключается к backend (контракты)

Ожидаемые endpoints:

- `GET /routines/today` — список активных рутины на выбранную дату (или “сегодня”)
- `POST /routines` — создание новой рутины
- `PATCH /routines/{id}/complete` — отметка выполнения (числовое/boolean значение + опциональные proofs)

Примечание по “proofs”:

- в будущем завершение может поддерживать гео-доказательство/фото
- UI должен показывать/запрашивать разрешения только при необходимости и с приватностью по умолчанию
